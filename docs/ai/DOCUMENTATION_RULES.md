```md
# Documentation Rules

## Purpose

Documentation should preserve decisions that matter later.

It should not describe every small implementation detail.

## Update documentation when a change affects

- architecture
- FSD structure
- API contract usage
- project setup
- dependency decisions
- important frontend conventions
- long-term technical decisions

## Do not update documentation for

- tiny visual changes
- isolated bug fixes
- temporary experiments
- internal implementation details with no future value

## Style

Documentation should be:

- short
- practical
- current
- connected to real project decisions

Avoid duplicating the same rule across many files.

Use links or references instead of copying the same text into several documents.
```
